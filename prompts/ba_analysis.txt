Analyse the following Jira ticket and produce a structured, developer-ready implementation specification.

JIRA TICKET:
ID: {ticket_id}
Summary: {ticket_summary}
Labels: {ticket_labels}

Description:
{ticket_description}

Existing Comments:
{ticket_comments}

AVAILABLE REPOSITORIES (key: repo — keywords that indicate changes belong here):
{available_repos}

ACTUAL FILES IN EACH REPOSITORY (every path here is real — use these to understand what already exists):
{repo_structures}

INSTRUCTIONS:
1. Identify the core functional requirement from all available fields — summary, description, comments, and labels.
2. Break the requirement into discrete, testable acceptance criteria (Given/When/Then format). Each criterion must be independently verifiable.
3. Determine which repos need changes. Read the file list carefully:
   - If the ticket adds or changes DATA (a new record, entry, or content), look at which repo owns the data layer. If one repo has a service, controller, or model layer (e.g. BlogService.java, a database, a seed file), changes to stored data MUST go in that repo — not the UI.
   - A UI repo that fetches data from an API does NOT own that data. Adding a new item served by the API requires changing the API repo.
   - Set "target_repos" to ALL repos that need changes. Missing a repo here means the developer will never touch it.
4. For each target repo, describe what needs to change in plain English under "repo_changes". Do NOT list specific file paths — the developer will read the codebase and decide which files to touch. Describe the functional area and what needs to happen (e.g. "Add a new blog entry to the seed data layer in BlogService", "Update the page background colour in the blogs page styling").
5. Use the file list to inform your understanding of the codebase (e.g. "the API uses Java Spring Boot with a BlogService", "the UI has a BlogCard component and fetches from the API") so your acceptance criteria and notes are accurate. But do not copy file paths into the output.
6. Specify any key implementation constraints the developer must know (e.g. "blog data comes from the API's BlogService seed data, not hardcoded in the UI").
7. Identify edge cases the developer must handle explicitly.
8. Estimate complexity: "low" (< 2h), "medium" (2–8h), "high" (> 8h).
9. CLARIFICATION RULES — only ask if you genuinely cannot proceed without the answer:
   - NEVER ask about things visible from the file list (what components exist, what patterns are used).
   - NEVER ask about conventions already established in the codebase — infer them.
   - NEVER ask about position/ordering of new items — default to appending at the end.
   - NEVER ask whether to reuse existing components — always reuse what already exists.
   - NEVER ask about fields not mentioned in the ticket — use only what the ticket specifies.
   - MUST ASK when the ticket says "update X" (colour, size, text, URL, etc.) without giving the new value — there is no way to implement this correctly without knowing the value. Do not invent or default it.
   - ONLY ask when a specific required value is absent from the ticket and cannot be inferred from context.
   - Aim for zero clarification questions. One is acceptable. More than two is a failure.

OUTPUT FORMAT:
Respond with a single JSON object only. No text outside the JSON.

```json
{{
  "ticket_id": "{ticket_id}",
  "needs_clarification": false,
  "clarification_questions": [],
  "summary": "One-sentence description of what needs to be built",
  "acceptance_criteria": [
    "Given X, when Y, then Z",
    "Given A, when B, then C"
  ],
  "target_repos": ["api", "ui"],
  "repo_changes": {{
    "api": "Describe what needs to change in the api repo in plain English",
    "ui": "Describe what needs to change in the ui repo in plain English"
  }},
  "edge_cases": [
    "Edge case the developer must handle"
  ],
  "complexity": "low",
  "implementation_notes": "Key constraints and context the developer needs — e.g. where data comes from, what must not be broken"
}}
```
